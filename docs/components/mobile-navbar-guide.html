Mobile Navbar Component

A responsive navigation component that adapts to different screen sizes. On small and medium screens, it provides a toggleable menu with fullscreen overlay. On desktop screens, navigation links are always visible.

Overview

The Mobile Navbar intelligently handles responsive navigation, providing a hamburger menu on mobile/tablet and traditional navigation links on desktop. The menu button provides visual feedback when the menu is open.

Basic Usage

Add the following HTML to your page header:

<header class="main" data-component="mobile-navbar">
    <nav role="navigation" aria-label="Main Navigation">
        <a href="/" class="brand">Brand</a>
        
        <ul class="links">
            <li><a href="#section1">Section 1</a></li>
            <li><a href="#section2">Section 2</a></li>
            <li><a href="#section3">Section 3</a></li>
        </ul>
        
        <div class="actions">
            <button class="menu toggle" aria-label="Toggle Menu">
                <span class="axicon render" data-name="Menu"></span>
            </button>
        </div>
    </nav>
</header>

JavaScript Initialization

Initialize the component with a module script:

<script type="module">
    import mobileNavbar from '../../js/components/mobile-navbar.js';
    const navElement = document.querySelector('header[data-component="mobile-navbar"]');
    if (navElement) {
        mobileNavbar.init(navElement);
    }
</script>

Responsive Behavior

  • Desktop (1024px+): Navigation links are always visible. Menu button still available.
  • Tablet & Mobile (<1024px): Navigation links hidden, menu button shows. Menu opens as fullscreen overlay.
  • Menu Button State: Shows primary background color when menu is open (visual feedback)

Features

  • Responsive: Automatically adapts to screen size
  • Fullscreen Menu: Menu takes full viewport height on mobile/tablet
  • Visual Feedback: Menu button changes color when menu is open
  • Accessible: Full ARIA support for screen readers
  • Keyboard Navigation: Press ESC to close menu, Tab to navigate
  • Outside Click: Menu closes when clicking outside
  • Link Click: Menu auto-closes when a link is clicked (SPA behavior)

Menu States

Closed State (Default)

Menu button shows no background, links are hidden on small screens.

Open State

Menu button shows primary color background as indicator. Navigation links display as fullscreen overlay. Menu can be closed by:

  • Clicking the menu button again
  • Clicking a link inside the menu
  • Clicking outside the menu
  • Pressing the ESC key

CSS Classes

Key classes used in the Mobile Navbar component:

  • .main - Main header wrapper (sticky)
  • .menu.toggle - Menu toggle button
  • .menu-open - Applied to header when menu is open
  • .links - Navigation links container
  • .actions - Action buttons container (menu, theme, search)

Design Tokens Used

  • var(--a-color-surface) - Menu background
  • var(--a-color-primary-container) - Menu button active background
  • var(--a-color-primary) - Menu button active color
  • var(--a-color-outline) - Menu border
  • var(--a-shadow-large) - Menu shadow

Accessibility Features

  • ARIA Controls: aria-controls links button to menu
  • ARIA Expanded: aria-expanded indicates menu state
  • ARIA Hidden: aria-hidden hides closed menu from screen readers
  • Keyboard Support: ESC key closes menu
  • Semantic HTML: Proper nav and button elements

Anatomy

The Mobile Navbar consists of the following parts:

  • Brand/Logo: Left-aligned branding element
  • Navigation Links: Main navigation items (hidden on mobile)
  • Actions: Search, GitHub, Theme, and Menu buttons
  • Menu Button: Hamburger icon (visible on mobile/tablet)
  • Fullscreen Menu: Overlay menu that appears on small screens

When to Use Mobile Navbar

Do Use

  • For responsive site headers and navigation
  • When supporting mobile, tablet, and desktop layouts
  • To provide consistent navigation across devices
  • When you need accessibility built-in
  • For SPA or multi-page applications

Don't Use

  • Don't use if you don't need responsive design
  • Avoid duplicating menu content unnecessarily
  • Don't hardcode breakpoints in your app
  • Avoid ignoring keyboard navigation needs
  • Don't skip testing on actual devices

Consider

  • Menu animation performance on slower devices
  • Keyboard focus management when menu opens/closes
  • Deep linking with menu state persistence
  • Mobile gesture support (swipe to open/close)
  • Integration with routing libraries

Browser Support

  • Chrome 90+
  • Firefox 88+
  • Safari 14+
  • Edge 90+
