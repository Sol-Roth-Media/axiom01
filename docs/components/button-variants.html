Button Variants & Styling

Comprehensive button component system with multiple variants, sizes, and states for consistent UI across your application. Session 2 Backport

Overview

Axiom provides a comprehensive button styling system built on semantic HTML. All buttons use the native <button> element with optional classes for variants and sizes.

Button Variants

Primary Buttons

Use for primary actions and main CTAs.

<button class="button primary">Primary Button</button>

Secondary Buttons

Use for secondary actions.

<button class="button secondary">Secondary Button</button>

Danger Buttons

Use for destructive actions (delete, remove, etc.).

<button class="button danger">Delete</button>

Success Buttons

Use for confirmatory actions.

<button class="button success">Confirm</button>

Icon-Only Buttons

Use for compact, icon-only interactions.

<button class="button icon-only" aria-label="Menu">
  <i class="fas fa-bars"></i>
</button>

Button Sizes

Small Buttons

<button class="button small">Small</button>

Medium Buttons (Default)

<button class="button">Medium</button>

Large Buttons

<button class="button large">Large</button>

Special Cases

Full-Width Buttons

Use in forms and modal dialogs where full width is needed.

<button class="button full-width">Full-Width</button>

Button Groups

Group related buttons together.

<div class="button-group">
  <button>Option 1</button>
  <button>Option 2</button>
  <button>Option 3</button>
</div>

States

Hover State

All buttons have smooth hover effects (opacity change + lift with shadow).

Active State

Applied during click/press, removes the lift effect.

Focus State

Visible outline for keyboard navigation and accessibility.

Disabled State

Use the disabled attribute. Button becomes 50% opacity and is not clickable.

CSS Variables

Buttons use Axiom's color system variables:

/* Color variables */
--a-color-primary           /* Primary button background */
--a-color-on-primary        /* Primary button text */
--a-color-secondary         /* Secondary button background */
--a-color-on-secondary      /* Secondary button text */
--a-color-error             /* Danger button background */
--a-color-on-error          /* Danger button text */
--a-color-success           /* Success button background */
--a-color-on-success        /* Success button text */

Accessibility

  • Semantic <button> elements
  • Visible focus states
  • Proper disabled state handling
  • Icon buttons have aria-label
  • High contrast in light and dark modes
  • Touch-friendly minimum size (44x44px)

Best Practices

  • Use semantically meaningful button text
  • Provide aria-label for icon-only buttons
  • Use danger buttons sparingly and clearly
  • Avoid using too many colors in one interface
  • Use appropriate button sizes for the context

Anatomy

Container

Wraps component elements and provides structure, spacing, and styling consistency.

Content Area

Primary content section where information is displayed or input is collected.

Interactive Parts

Buttons, toggles, or controls. Always keyboard accessible and touch-friendly.

States & Variants

Different states (normal, hover, active, disabled) and size/color variants supported.

When to Use Button Variants

Do Use

  • Use this component for its intended purpose
  • Follow the semantic structure
  • Implement proper accessibility
  • Respect responsive design principles
  • Use design tokens for customization

Don't Use

  • Don't modify core structure unnecessarily
  • Avoid hardcoding styles or colors
  • Don't skip accessibility features
  • Avoid responsive design gotchas
  • Don't duplicate when a component exists

Consider

  • Your specific use case and context
  • User accessibility needs (WCAG AAA)
  • Dark mode and theme variations
  • Mobile and desktop experiences
  • Performance optimization
