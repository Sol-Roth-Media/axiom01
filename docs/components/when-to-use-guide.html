When to Use Cards

A reusable card component for documenting best practices, anti-patterns, and considerations. Perfect for design systems, component libraries, and documentation.

Overview

The When to Use cards provide a visual way to communicate design decisions and guidelines. Use them in your documentation to help developers and designers understand when and how to use components.

Basic Usage

Add the following HTML to your documentation:

<div class="when-to-use-cards">
    <div class="when-to-use-card do">
        <h3><span class="icon">✓</span> Do Use</h3>
        <ul>
            <li>Use this component when appropriate</li>
            <li>Provides clear semantic meaning</li>
        </ul>
    </div>
    
    <div class="when-to-use-card dont">
        <h3><span class="icon">✗</span> Don't Use</h3>
        <ul>
            <li>Don't use for unrelated content</li>
            <li>Avoid hardcoding styles</li>
        </ul>
    </div>
    
    <div class="when-to-use-card consider">
        <h3><span class="icon">⚠</span> Consider</h3>
        <ul>
            <li>Think about accessibility requirements</li>
            <li>Check responsive behavior</li>
        </ul>
    </div>
</div>

Card Types

Do Use (Green)

Use the "do" class for recommended practices and best uses of your component.

✓ Do Use

  • For component recommendations
  • Best practice examples
  • Encouraged usage patterns

Don't Use (Red)

Use the "dont" class for anti-patterns and things to avoid.

✗ Don't Use

  • For anti-patterns
  • Common mistakes to avoid
  • What NOT to do

Consider (Orange/Yellow)

Use the "consider" class for considerations and special cases.

⚠ Consider

  • For edge cases
  • Accessibility requirements
  • Things to keep in mind

All Three Together

Here's how all three card types look together:

✓ Do Use

  • Use semantic HTML elements
  • Follow design system tokens
  • Test accessibility
  • Support responsive design

✗ Don't Use

  • Non-semantic markup
  • Hardcoded colors or spacing
  • Ignore accessibility guidelines
  • Skip responsive testing

⚠ Consider

  • Browser compatibility needs
  • Performance implications
  • Theme customization
  • Mobile-first approach

CSS Classes

    gap: var(--a-space-l);
    margin: var(--a-space-l) 0;

    padding: var(--a-space-l);
    border: 2px solid;

Design Tokens Used

  • var(--a-color-success) - Green borders
  • var(--a-color-error) - Red borders
  • var(--a-color-warning) - Yellow/Orange borders
  • var(--a-space-l) - Padding and gaps
  • var(--a-border-radius-large) - Border radius
