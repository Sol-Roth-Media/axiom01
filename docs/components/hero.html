Hero Component

Mobile-optimized hero sections with responsive typography, button sizing, and optional touch-swipe carousel functionality for rich content presentation.

Hero Component Features

The hero component is the primary landing area of a page. Axiom's hero is designed with mobile-first principles, automatically adjusting typography, button sizes, and layout for different screen sizes. It now includes optional swipe carousel functionality for presenting multiple content panels.

Features

Mobile-First Design

Responsive Typography

Headings scale from 1.5rem on mobile to 3.2rem on desktop

Smart Button Sizing

Buttons automatically reduce from lg to md on mobile to prevent stacking

Theme Compatible

Mobile Optimizations

The hero component automatically applies mobile-specific styles at different breakpoints:

Breakpoint H1 Size Tagline Size Button Layout Spacing
Desktop (1024px+) 3.2rem 1.25rem Flex row, inline Generous
Tablet (768px-1023px) 2.5rem 1.1rem Flex row, may wrap Medium
Mobile (480px-767px) 1.875rem 0.95rem Flex column, full width Compact
Small Mobile (< 480px) 1.5rem 0.9rem Flex column, full width Minimal

HTML Structure

Basic Hero

<section class="hero">
  <div class="content">
    <h1>Your Headline</h1>
    <p class="tagline">Optional subtitle or tagline</p>
    <p>Longer descriptive paragraph...</p>
    <div class="actions">
      <a href="#" class="button primary lg">CTA Button</a>
      <a href="#" class="button secondary lg">Secondary Action</a>
    </div>
  </div>
</section>

Swipeable Hero Carousel

For touch-enabled interfaces, the hero can support horizontal swiping to switch between content panels:

Swipeable Hero HTML

<section class="hero swipeable" data-swipeable>
  <div class="swipe-panel">
    <h1>Panel 1</h1>
    <p>Swipe left to see more...</p>
  </div>
  <div class="swipe-panel">
    <h1>Panel 2</h1>
    <p>Second content panel</p>
  </div>
  <div class="swipe-panel">
    <h1>Panel 3</h1>
    <p>Third content panel</p>
  </div>
</section>

JavaScript API

// Auto-initialization (on data-swipeable elements)
// Happens on DOMContentLoaded

// Manual initialization
Axiom.initHeroSwipe(element)

// Listen to swipe events
element.addEventListener('hero-swipe', (e) => {
  console.log(e.detail.direction) // 'left' or 'right'
})

// Component tracking
Axiom.initializedComponents.has(element)
Axiom.destroyComponent(element)

CSS Classes

Class Element Purpose
.hero <section> Main hero container
.hero.small <section> Smaller hero variant (documentation pages)
.hero.swipeable <section> Enable horizontal scroll/swipe
.content <div> Content wrapper
.tagline <p> Subtitle/tagline paragraph
.actions <div> Button group container
.swipe-panel <div> Individual carousel panel
.bem-tooltip <abbr> Tooltip for term definitions

Responsive Button Behavior

  • Desktop: Buttons appear inline (flex-row) in action bar
  • Tablet: Buttons may wrap if space is limited
  • Mobile: Buttons stack vertically (flex-column) and take full width
  • Mobile button sizing: Automatically reduced padding and font-size to prevent excessive height

BEM Tooltip Explanation

Hover over or click the "BEM" text in hero examples to see a tooltip explaining what BEM (Block Element Modifier) is. This is a demonstration of Axiom's accessibility and information-rich design:

<abbr class="bem-tooltip" 
      data-tooltip="Block Element Modifier - a complex naming convention..."
      title="BEM">BEM</abbr>

Accessibility

  • <section> provides semantic structure
  • <h1> establishes page hierarchy
  • Buttons are semantic <button> or <a> elements
  • Tooltips use <abbr> with title attribute for screen readers
  • Swipe gestures have keyboard fallback (not swipe-only)
  • High color contrast maintained in all themes

Performance Tips

  • Use optimized images or gradients for backgrounds
  • Keep text concise for better mobile display
  • Limit buttons to 2-3 actions to avoid overwhelming users
  • Test on actual mobile devices (especially hero swipe)
  • Use loading="lazy" for images below the fold

Browser Support

All modern browsers. Swipe support requires touch-enabled devices (phones, tablets) or mouse simulation in dev tools.
